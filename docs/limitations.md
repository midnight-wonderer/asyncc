# Limitations and Pitfalls

While `asyncc` provides a highly efficient async/await abstraction for C with zero runtime memory allocations, it relies on compile-time source preprocessing and macro state-saving. Because of these design choices, developers must adhere to several rules and restrictions.

---

## 1. The Switch-Case Constraint (Duff's Device)

`asyncc` implements coroutines by wrapping the function body in a `switch` statement (via `asyncc_begin` and `asyncc_end`) and mapping yield/await locations to `case __LINE__:` labels.

### The Pitfall
You **cannot** use a standard C `switch` statement inside a coroutine if that switch statement contains any yield or await macros (`asyncc_yield`, `await`, `asyncc_chan_read`, `asyncc_chan_write`, etc.).

If you attempt this:
```c
asyncc asyncc_state_t bad_task(int state)
{
    asyncc_begin;

    switch (state) {
        case 1:
            // CRITICAL ERROR: asyncc_yield cannot be used inside a local switch
            asyncc_yield; 
            break;
        case 2:
            do_something();
            break;
    }

    asyncc_end;
}
```
The compiler will fail with syntax errors because the `case` label generated by `asyncc_yield` is nested inside your local `switch` rather than the main coroutine `switch`.

### Workarounds
1. **Use `if-else` chains** instead of `switch` statements if you need to yield inside the branches.
2. **Move state logic to helper functions** that do not yield.

---

## 2. Local Variable Declaration Scope

To make local variables persist across function returns and yields, they must be parsed by the Python preprocessor and added to the generated state structure.

### The Pitfall
Any variable declared **after** `asyncc_begin` is treated as a standard C stack variable. Its value will be lost when the coroutine yields and resumes.

```c
asyncc asyncc_state_t bad_locals(void)
{
    // Good: This variable is declared before asyncc_begin and persists
    int persistent_var;

    asyncc_begin;
    
    // Bad: This variable is declared after asyncc_begin
    int temporary_var = 10;
    
    asyncc_yield;
    
    // CRITICAL BUG: temporary_var's value is now undefined/garbage!
    printf("%d\n", temporary_var); 

    asyncc_end;
}
```

### Workaround
Always declare all local variables that need to retain their values across yields/awaits at the top of the function, **before** `asyncc_begin`.

---

## 3. Variable Name Shadowing

The Python preprocessor performs a simple AST traversal and replaces all references to parameters and top-level local variables with their struct member representation (`l->variable`).

### The Pitfall
If you declare a variable in a inner nested scope (e.g. inside an `if` block, a `for` loop, or a helper macro) that matches the name of a parameter or a variable declared before `asyncc_begin`, the preprocessor will rewrite the inner reference to `l->variable` too.

```c
asyncc asyncc_state_t shadowing_task(int value)
{
    asyncc_begin;
    
    if (cond) {
        // Shadowing: value is a local inside the block
        int value = 5;
        // The preprocessor will rewrite this to: l->value = 5,
        // overwriting the parameter l->value!
    }

    asyncc_end;
}
```

### Workaround
Use unique, descriptive names for all variables within a coroutine, and avoid name shadowing in nested blocks.

---

## 4. Single-Reader / Single-Writer Channels

The CSP channels (`asyncc_chan_t`) in `asyncc` are optimized for speed and footprint. The channel structure tracks block status using a single reader pointer (`waiting_reader`) and a single writer pointer (`waiting_writer`).

### The Pitfall
* Only **one** coroutine can block waiting to read from a specific channel at a time.
* Only **one** coroutine can block waiting to write to a specific channel at a time.

If multiple coroutines try to call `asyncc_chan_read` or `asyncc_chan_write` on the same channel simultaneously:
1. The second coroutine will overwrite the channel's wait pointer (`waiting_reader` or `waiting_writer`).
2. The first coroutine will remain blocked forever (lost wakeup / deadlock).

### Workaround
Restrict each channel to a **single producer and a single consumer**. 
If you need multi-producer or multi-consumer patterns:
- Use separate channels for each task.
- Create a dedicated coordinator/multiplexer coroutine that reads from multiple channels and forwards data to a single channel.

---

## 5. Thread Safety

`asyncc` is designed for cooperative multitasking within a single execution thread.

### The Pitfall
The runner scheduler (`asyncc_runner_t`) and channels (`asyncc_chan_t`) are **not** thread-safe. Modifying them from different OS threads or hardware interrupts without external synchronization will cause memory corruption.

### Workarounds
1. **Single-threaded Event Loops**: Run each `asyncc_runner_t` inside its own dedicated OS thread or main loop.
2. **Inter-Thread Communication**: Use `asyncc_gate_t` to bridge threads. External threads or ISRs can safely call `asyncc_gate_signal` to wake up tasks inside the runner.
